## 1. Auth & Identity

### 1.1 Email/Password Signup

**One-line purpose:** Create a new user account with email and password, returning dual JWT tokens.

**User-facing trigger:** User fills in the register form at `/register` and submits.

**Full file chain (in execution order):**
1. `frontend/app/(auth)/register/page.tsx` — collects name, email, password; posts to FastAPI directly
2. `frontend/lib/auth.ts` — `signIn("jwt", {...})` stores the returned tokens in NextAuth session cookie
3. `backend/routers/auth.py` — `register()` handler validates uniqueness, checks `ADMIN_EMAILS`, hashes password
4. `backend/models/user.py` — `User` ORM model persisted to `users` table
5. PostgreSQL (`users` table) — row inserted with `bcrypt` hash, `provider='email'`, `role='user'` (or `'admin'`)
6. Response flows back: access + refresh tokens → NextAuth stores in encrypted cookie → client receives session

**Key functions/methods involved:**
- `register()` in `backend/routers/auth.py`
- `_hash_password(password)` — `bcrypt.hashpw()` directly (not passlib)
- `_create_access_token(user_id)` — 15-minute JWT
- `_create_refresh_token(user_id)` — 7-day JWT
- `signIn("jwt", credentials)` in NextAuth

**Database tables read/written:** `users` (write)

**External services called:** None

**Key technical decisions:**
- `bcrypt` is used directly (not `passlib.CryptContext`) because `passlib` has a bug with `bcrypt >= 4.x` that causes 500 errors on password hashing.
- Admin role is granted automatically at registration if the email appears in the `ADMIN_EMAILS` environment variable (comma-separated list).
- `password_hash` is nullable by design to support Google-only accounts.

**Failure modes / edge cases handled:**
- Duplicate email → HTTP 409 Conflict.
- Missing `ADMIN_EMAILS` env var → falls back to `ADMIN_EMAIL` single-value var for backward compatibility.

**Dependencies on other features:** Feeds into all authenticated features.

### 1.2 Email/Password Login (Dual JWT)

**One-line purpose:** Authenticate with email + password and issue a short-lived access token plus a long-lived refresh token.

**User-facing trigger:** User enters credentials on `/login` page.

**Full file chain (in execution order):**
1. `frontend/app/(auth)/login/page.tsx` — posts credentials directly to FastAPI `/api/auth/login`
2. `backend/routers/auth.py` — `login()` queries user by email, verifies bcrypt hash, checks `is_active`
3. PostgreSQL (`users` table) — user row read
4. `backend/routers/auth.py` — `_create_access_token()` + `_create_refresh_token()` issued
5. `frontend/lib/auth.ts` — `signIn("jwt", { accessToken, refreshToken, ... })` stores in NextAuth cookie
6. `frontend/middleware.ts` — protects all subsequent `/dashboard/*`, `/chatbot/*`, etc. routes

**Key functions/methods involved:**
- `login()` in `backend/routers/auth.py`
- `_verify_password(plain, hashed)` — `bcrypt.checkpw()`
- `_create_access_token()`, `_create_refresh_token()`
- `authorize()` callback in `frontend/lib/auth.ts` (NextAuth Credentials provider)

**Database tables read/written:** `users` (read)

**External services called:** None

**Key technical decisions:**
- Dual JWT (access 15 min + refresh 7 days): access token is short-lived for security; refresh token enables silent renewal without re-login.
- If `password_hash IS NULL` (Google-only account), login returns HTTP 400 with detail `"no_password_set"` — the frontend then prompts the user to sign in with Google.
- The access token expire time is set to `ACCESS_TOKEN_EXPIRE_MINUTES` (default 15, but `frontend/lib/auth.ts` treats it as 30 minutes based on the `ACCESS_TOKEN_TTL_MS` constant, refreshing 60s before expiry).

**Failure modes / edge cases handled:**
- Deactivated account → HTTP 401 `"Account is disabled"`.
- Wrong password → generic HTTP 401 (no enumeration of which field was wrong).
- Google account without password → HTTP 400 `"no_password_set"`.

**Dependencies on other features:** Silent token refresh (1.4), all authenticated API calls via `frontend/lib/api.ts`.

### 1.3 Google OAuth Login

**One-line purpose:** Verify a Google ID token from the frontend and issue BahasaBot's own JWT tokens for the account.

**User-facing trigger:** User clicks "Continue with Google" on login or register page.

**Full file chain (in execution order):**
1. `frontend/app/(auth)/login/page.tsx` — Google Identity Services button triggers client-side popup, receives `id_token`
2. Frontend posts `id_token` to `POST /api/auth/google`
3. `backend/routers/auth.py` — `google_auth()` calls `google.oauth2.id_token.verify_oauth2_token()` against Google's public keys
4. PostgreSQL (`users` table) — `SELECT` by email; if not found, `INSERT` with `password_hash=NULL`, `provider='google'`
5. BahasaBot's own JWT tokens issued; `requires_password_setup=True` if `password_hash IS NULL`
6. `frontend/lib/auth.ts` — `signIn("jwt", ...)` stores tokens; frontend detects `requires_password_setup` flag
7. `frontend/components/auth/SetPasswordModal.tsx` — non-dismissible modal shown if password not yet set

**Key functions/methods involved:**
- `google_auth()` in `backend/routers/auth.py`
- `google_id_token.verify_oauth2_token()` from `google.oauth2`
- `SetPasswordModal` component in `frontend/components/auth/SetPasswordModal.tsx`
- `authApi.setPassword()` in `frontend/lib/api.ts`

**Database tables read/written:** `users` (read + conditional write)

**External services called:** Google's public key endpoint (for token verification via `google-auth` library)

**Key technical decisions:**
- BahasaBot never stores or handles Google OAuth credentials — it only verifies the ID token, then issues its own JWT. This keeps the auth flow decoupled from Google.
- If an existing email/password account signs in via Google, `provider` is updated to `'google'` on first OAuth sign-in, linking the accounts.
- `google-auth` is pinned to `2.29.0` to avoid version conflicts with `google-generativeai`.

**Failure modes / edge cases handled:**
- Unverified Google email → HTTP 401.
- `GOOGLE_CLIENT_ID` not set → HTTP 500 with descriptive error.
- Deactivated account → HTTP 401.

**Dependencies on other features:** Set Password modal (1.7), all authenticated features.

### 1.4 Silent Token Refresh

**One-line purpose:** Automatically renew the access token using the refresh token when it expires, without user interaction.

**User-facing trigger:** Background/automatic — triggered when `Date.now() >= accessTokenExpires` during any API call.

**Full file chain (in execution order):**
1. `frontend/lib/auth.ts` — NextAuth `jwt` callback runs on every session check; compares `accessTokenExpires` vs `Date.now()`
2. `frontend/lib/auth.ts` — `refreshAccessToken(refreshToken)` calls `POST /api/auth/refresh`
3. `backend/routers/auth.py` — `refresh_token()` decodes the refresh JWT, verifies user still active, issues new access token
4. `frontend/lib/auth.ts` — new `accessToken` + updated `accessTokenExpires` stored back in the NextAuth token
5. `frontend/lib/api.ts` — `getCachedSession()` with 1-minute session cache deduplicates concurrent refresh calls

**Key functions/methods involved:**
- `refreshAccessToken()` in `frontend/lib/auth.ts`
- `refresh_token()` in `backend/routers/auth.py`
- `getCachedSession()` in `frontend/lib/api.ts`
- `invalidateSessionCache()` in `frontend/lib/api.ts`

**Database tables read/written:** `users` (read, to verify `is_active`)

**External services called:** None

**Key technical decisions:**
- Session cache (`SESSION_CACHE_TTL_MS = 60_000` ms) deduplicates `getSession()` calls from multiple concurrently mounted components (e.g., `NotificationBell` + sidebar streak badge) so only one `/api/auth/session` request fires per minute.
- An in-flight deduplication promise (`_sessionInflight`) further prevents N parallel `getSession()` calls from the same mount.
- If refresh fails, token gets `error: "RefreshTokenExpired"` — `api.ts` interceptor then calls `signOut()` automatically.

**Failure modes / edge cases handled:**
- Refresh token itself expired → user is signed out via `signOut({ callbackUrl: "/login" })`.
- Network error during refresh → user is signed out.

**Dependencies on other features:** All authenticated API calls.

### 1.5 Forgot Password (6-digit code flow)

**One-line purpose:** Send a 6-digit time-limited verification code to the user's email for password reset.

**User-facing trigger:** User clicks "Forgot Password" on `/forgot-password` page and enters their email.

**Full file chain (in execution order):**
1. `frontend/app/(auth)/forgot-password/page.tsx` — collects email, posts to `/api/auth/forgot-password`
2. `backend/routers/auth.py` — `forgot_password()` looks up user by email; if Google account, returns 400
3. Invalidates any existing unused codes for this user
4. `secrets.randbelow(1_000_000)` generates 6-digit code; `_make_code_hash(email, code)` = SHA-256(`email:code`)
5. `backend/models/password_reset.py` — `PasswordResetToken` row inserted with `expires_at = now + 10 min`
6. `backend/services/email_service.py` — `send_reset_email()` → `asyncio.to_thread(_send_code_email_sync)` → Resend SDK
7. Frontend advances user to code-entry step

**Key functions/methods involved:**
- `forgot_password()` in `backend/routers/auth.py`
- `_make_code_hash(email, code)` — SHA-256 of `"email:code"` string
- `send_reset_email(to_email, code)` in `backend/services/email_service.py`
- `_send_code_email_sync(to_email, code)` — Resend SDK (sync, run in thread)

**Database tables read/written:** `users` (read), `password_reset_tokens` (write)

**External services called:** Resend (email delivery)

**Key technical decisions:**
- SHA-256 hash of `"email:code"` ties the code to a specific email — prevents one user's code being used for another account.
- Non-existent email returns HTTP 200 with a generic message to prevent email enumeration.
- `send_reset_email` is fire-and-forget with error logging — if Resend fails, the code is still stored in the DB (manual recovery possible).
- The Resend SDK is synchronously called inside `asyncio.to_thread()` so it doesn't block the event loop.

**Failure modes / edge cases handled:**
- Google account → HTTP 400 with detail `"google_account_no_password"`.
- Resend delivery failure → logs warning, returns success to user (prevents timing oracle).
- `RESEND_API_KEY` not set → logs error, returns success to user.

**Dependencies on other features:** Reset Password (1.6).

### 1.6 Reset Password

**One-line purpose:** Consume a verified 6-digit code and update the user's password hash.

**User-facing trigger:** User enters the 6-digit code received by email, then enters new password on `/reset-password` page.

**Full file chain (in execution order):**
1. `frontend/app/(auth)/reset-password/page.tsx` — 3-step flow: code entry → code verification → new password
2. Step 1: `POST /api/auth/verify-reset-code` — validates code is not expired/used (does NOT consume it)
3. Step 2: `POST /api/auth/reset-password` — re-validates code, updates `password_hash`, marks `used=True`
4. `backend/routers/auth.py` — `reset_password()` atomically updates hash and invalidates code

**Key functions/methods involved:**
- `verify_reset_code()` in `backend/routers/auth.py` (non-consuming check)
- `reset_password()` in `backend/routers/auth.py` (consuming, atomic update)
- `_make_code_hash(email, code)` — same hash function used to look up the stored token

**Database tables read/written:** `password_reset_tokens` (read + write), `users` (write)

**External services called:** None

**Key technical decisions:**
- Two-step design: verify first (non-consuming) so frontend can advance to password step confidently; consume on final submit. Both use the same hash lookup, so the code cannot be used twice.
- 10-minute TTL (`RESET_CODE_TTL_MINUTES = 10`) is enforced by comparing `expires_at` against `datetime.now(utc)`. Timezone-naive timestamps from DB are converted to UTC before comparison.

**Failure modes / edge cases handled:**
- Expired code → HTTP 400.
- Already-used code → HTTP 400.
- Non-existent token hash → HTTP 400.

**Dependencies on other features:** Forgot Password (1.5).

### 1.7 Set Password for Google Users

**One-line purpose:** Allow a Google-only account to set a password so they can log in via email in the future.

**User-facing trigger:** Shown immediately after first Google sign-in when `password_hash IS NULL` — the `SetPasswordModal` is non-dismissible.

**Full file chain (in execution order):**
1. `frontend/app/(dashboard)/layout.tsx` or login page — detects `requires_password_setup` flag in token response
2. `frontend/components/auth/SetPasswordModal.tsx` — full-screen z-[90] backdrop, two-field form, no dismiss option
3. `authApi.setPassword(new_password)` → `POST /api/auth/set-password`
4. `backend/routers/auth.py` — `set_password()` verifies `password_hash IS NULL`, bcrypt-hashes, saves
5. Frontend redirects to `/dashboard` after success

**Key functions/methods involved:**
- `set_password()` in `backend/routers/auth.py`
- `SetPasswordModal` component in `frontend/components/auth/SetPasswordModal.tsx`
- `authApi.setPassword()` in `frontend/lib/api.ts`

**Database tables read/written:** `users` (write)

**External services called:** None

**Key technical decisions:**
- Endpoint is guarded: if `password_hash` is already set, returns HTTP 400 — prevents overwriting via this endpoint (use `change-password` instead).
- Modal is non-dismissible by design — Google users must set a password to use the platform.
- z-index 90 places it above the sidebar (z-70) and onboarding modal (z-80).

**Failure modes / edge cases handled:**
- Edge case where password was set in another browser tab — frontend detects `"already has a password"` detail and continues to dashboard.

**Dependencies on other features:** Google OAuth Login (1.3), Profile (5.1).

### 1.8 Logout / Session Invalidation

**One-line purpose:** Clear the NextAuth session cookie and redirect to login.

**User-facing trigger:** User clicks "Sign out" in the sidebar or settings.

**Full file chain (in execution order):**
1. Sidebar / settings button calls `signOut({ callbackUrl: "/login" })` from `next-auth/react`
2. `frontend/lib/api.ts` — `invalidateSessionCache()` clears the 1-minute session cache
3. NextAuth clears the encrypted session cookie
4. User is redirected to `/login`

**Key functions/methods involved:**
- `signOut()` from `next-auth/react`
- `invalidateSessionCache()` in `frontend/lib/api.ts`

**Database tables read/written:** None (JWTs are stateless — there is no server-side token revocation)

**External services called:** None

**Key technical decisions:**
- No server-side token blacklist — logout is client-side only. The refresh token remains valid for 7 days on the backend side, but with no client holding it this is low-risk for a 30-user platform.
- `api.ts` response interceptor auto-triggers `signOut()` on any HTTP 401, providing a consistent logout path.

**Failure modes / edge cases handled:**
- HTTP 401 from any API call triggers automatic logout.
- `RefreshTokenExpired` error in session triggers `signOut()` from the request interceptor.

**Dependencies on other features:** All authenticated features.

## 2. Onboarding

### 2.1 Multi-Step Onboarding Modal

**One-line purpose:** Collect new-user preferences (language, goal, level, timeline) and optionally generate a learning roadmap — shown once on first login.

**User-facing trigger:** Any authenticated page load when `onboarding_completed === false` on the user record.

**Full file chain (in execution order):**
1. `frontend/app/(dashboard)/layout.tsx` — checks `onboarding_completed` flag from session/profile; renders `OnboardingModal` if false
2. `frontend/components/onboarding/OnboardingModal.tsx` — 8-step flow (Welcome, Native Language, Why learning, BPS level, Goal, Timeline, Daily study time, Gender/Age for image personalisation)
3. On "Generate Roadmap" (step 7 primary action): `PATCH /api/profile/` with all collected data + `onboarding_completed=true`, then `POST /api/journey/roadmap/generate`
4. On "Skip": `PATCH /api/profile/` with collected data + `onboarding_completed=true` only
5. `backend/routers/profile.py` — `update_profile()` persists `native_language`, `learning_goal`, `onboarding_completed`, `gender`, `age_range`
6. `backend/routers/journey.py` — `generate_roadmap()` → `journey_service.generate_roadmap()`
7. After success, `frontend/components/onboarding/UITour.tsx` — spotlight tour of sidebar features shown when `has_seen_tour === false`

**Key functions/methods involved:**
- `OnboardingModal` in `frontend/components/onboarding/OnboardingModal.tsx`
- `OnboardingStep` in `frontend/components/onboarding/OnboardingStep.tsx`
- `UITour` in `frontend/components/onboarding/UITour.tsx`
- `profileApi.updateProfile()`, `journeyApi.generateRoadmap()` from `frontend/lib/api.ts`
- `update_profile()` in `backend/routers/profile.py`
- `generate_roadmap()` in `backend/services/journey_service.py`

**Database tables read/written:** `users` (write — `onboarding_completed`, `native_language`, `learning_goal`, `gender`, `age_range`, `has_seen_tour`), `user_roadmaps` (write, if roadmap generated)

**External services called:** Google Gemini (if roadmap generated — see Journey feature 4.2)

**Key technical decisions:**
- All steps are skippable — clicking skip calls `PATCH /api/profile/` with whatever data was collected so far, plus `onboarding_completed=true`. This ensures the modal never shows again even if the user skips every step.
- Journey goal text from onboarding is saved to `users.learning_goal` as free text, overwriting the generic onboarding dropdown choice. Journey roadmap generation later overwrites this again for even fresher personalization.
- `gender` and `age_range` are collected in onboarding specifically to personalize the Journey banner image generation prompt (which depicts a learner figure appropriate to the user's demographics).
- `has_seen_tour` is a separate boolean — the UI Tour (`UITour.tsx`) is gated on this, not `onboarding_completed`.

**Failure modes / edge cases handled:**
- Roadmap generation failure → calls `onComplete("done")` instead of `"roadmap_ready"`, so user reaches dashboard even if Gemini is down.
- Modal is non-dismissible (clicking backdrop does nothing).

**Dependencies on other features:** Profile (5.1), Journey (4.2), UI Tour.

## 3. Core Learning Flows

### 3.1 Chatbot Conversation (RAG + LangChain + SSE Streaming)

**One-line purpose:** AI tutor conversation powered by Gemini + RAG retrieval, streaming token-by-token via Server-Sent Events.

**User-facing trigger:** User types a message in the chatbot UI at `/chatbot` and clicks send.

**Full file chain (in execution order):**
1. `frontend/app/(dashboard)/chatbot/page.tsx` — manages EventSource / SSE connection, renders `ChatMessage` components
2. `frontend/components/chatbot/ChatMessage.tsx` — renders message bubbles; `VocabularyHighlight.tsx` highlights vocab pills
3. `frontend/lib/api.ts` — `chatbotApi.sendMessage()` opens SSE to `POST /api/chatbot/message`
4. `backend/routers/chatbot.py` — `send_message()` rate-limited at 20/min; resolves/creates session; returns `EventSourceResponse`
5. `backend/routers/chatbot.py` — `event_generator()` async generator yields `{"type": "ping"}`, then tokens, then `{"type": "done"}`
6. `backend/services/langchain_service.py` — `stream_chat_response()`:
   a. Saves user message to DB
   b. Parallel Redis lookups: RAG context cache, profile cache, history cache (`asyncio.gather`)
   c. DB fallbacks on cache miss
   d. Builds system prompt with RAG context + personalization strings
   e. Calls `gemini_service.stream_text()` → `ChatGoogleGenerativeAI.astream()`
   f. Saves assistant message to DB
   g. `asyncio.create_task(_extract_and_save(...))` — background vocab/grammar extraction
7. `backend/services/rag_service.py` — `similarity_search(query, k=3, db)` — cosine distance on `documents.embedding`
8. `backend/services/gemini_service.py` — `stream_text()` uses `_chatbot_llm().astream()` (ChatGoogleGenerativeAI)
9. Background: `_extract_and_save()` calls `extract_vocab_and_grammar()` → another Gemini call → saves to `vocabulary_learned` + `grammar_learned`
10. Background: `_award_chatbot_xp()` — awards 5 XP for first message of a session (Redis-gated per session_id)

**Key functions/methods involved:**
- `send_message()` in `backend/routers/chatbot.py`
- `stream_chat_response()` in `backend/services/langchain_service.py`
- `get_or_create_session()` in `backend/services/langchain_service.py`
- `_load_history()`, `_update_history_cache()` in `backend/services/langchain_service.py`
- `get_cached_profile()` in `backend/services/langchain_service.py`
- `similarity_search()` in `backend/services/rag_service.py`
- `stream_text()` in `backend/services/gemini_service.py`
- `extract_vocab_and_grammar()` in `backend/services/langchain_service.py`
- `_extract_and_save()` in `backend/services/langchain_service.py`
- `_award_chatbot_xp()` in `backend/routers/chatbot.py`
- `prewarm_chatbot()` — called on page mount to warm profile cache

**Database tables read/written:**
- `chat_sessions` (read + write), `chat_messages` (write × 2), `documents` (read — RAG), `vocabulary_learned` (write), `grammar_learned` (write), `users` (read — profile)

**External services called:** Google Gemini API (chatbot streaming + extraction)

**Key technical decisions:**
- Three Redis lookups run in parallel with `asyncio.gather()` to save ~10-15ms on the warm path.
- History window is capped at 6 messages (reduced from 10) to keep prompt size small and reduce TTFT.
- RAG uses k=3 (reduced from 5) for the same reason.
- Vocab/grammar extraction is a separate `asyncio.create_task()` — it never delays token delivery to the user.
- The chatbot system prompt (`CHATBOT_SYSTEM_PROMPT`) encodes full awareness of all other BahasaBot features and includes a redirect rule for off-topic requests (e.g., "give me a quiz" → redirected to Quiz feature with exact sidebar instructions).
- Language detection is built into the system prompt: if the user writes in English, the entire response must be in English (Malay appears only as examples). This is a hard rule that overrides all other instructions.

**Failure modes / edge cases handled:**
- Gemini streaming error → yields `{"type": "error", "message": "Something went wrong"}`.
- Session not found → HTTP 404 before stream starts.
- Redis unavailable → falls back to DB for history/profile.
- Vocab extraction failure → logged and silently swallowed (never disrupts the response).

**Dependencies on other features:** RAG corpus (must be seeded), Gamification (XP), Vocabulary tracking (4.4), Grammar tracking (4.5), Dashboard (feeds data).

### 3.2 Course Generation (Content Filter + Background Job)

**One-line purpose:** Generate a full structured AI course from a topic string, running as a background task with live progress polling.

**User-facing trigger:** User enters a topic in `CourseGenerationModal` and clicks Generate.

**Full file chain (in execution order):**

**Key technical decisions:**
- `asyncio.Semaphore(2)` limits concurrent class-generation Gemini calls to 2, respecting the free-tier 5 RPM limit. Without this, parallel calls would 429.

### 3.4 Module Quiz (Cache + Pass Gate + Module Unlock)

**One-line purpose:** Auto-generated 10-question quiz from module content; must pass (≥70%) to unlock next module.

**User-facing trigger:** All classes in a module completed — user clicks "Take Quiz" CTA.

**Full file chain (in execution order):**
1. `frontend/app/(dashboard)/courses/[courseId]/modules/[moduleId]/quiz/page.tsx` — renders quiz interface
2. `GET /api/courses/{courseId}/modules/{moduleId}/quiz` → `get_module_quiz()`
3. `backend/services/quiz_service.py` — `get_module_quiz()`: checks if already passed; serves from Redis cache `quiz:module:{module_id}:user:{user_id}` (TTL 2hr); else generates
4. `generate_module_quiz(module_id, db)` — fetches all classes, builds content summary, calls Gemini for 10 questions (6 MCQ + 4 fill-in-blank); validates structure
5. Questions returned to frontend WITHOUT `correct_answer` field
6. User submits answers → `POST /api/courses/{courseId}/modules/{moduleId}/quiz`
7. `backend/services/quiz_service.py` — `submit_module_quiz()`: loads questions WITH answers from Redis cache; scores; saves `ModuleQuizAttempt`; updates `weak_points` for wrong answers; if score ≥ 70%, inserts `UserProgress(class_id=None)` (module pass row); clears cache
8. If newly passed: `asyncio.create_task(_check_course_completion_for_journey())` — checks if all modules done, triggers journey roadmap progress hook
9. Frontend shows results page with per-question feedback; passes to next module

**Key functions/methods involved:**
- `get_module_quiz()`, `generate_module_quiz()`, `submit_module_quiz()` in `backend/services/quiz_service.py`
- `_update_weak_points()` in `backend/services/quiz_service.py`
- `_check_course_completion_for_journey()` in `backend/services/quiz_service.py`
- `_quiz_cache_key(module_id, user_id)` — Redis key pattern

**Database tables read/written:** `modules`, `classes` (read), `module_quiz_attempts` (write), `user_progress` (write), `weak_points` (write)

**External services called:** Google Gemini API (quiz generation)

**Key technical decisions:**
- `correct_answer` is never sent to the frontend — scoring is 100% server-side. Cache hit on submit is required; if expired between GET and POST, the quiz is regenerated for scoring purposes.
- PASS_THRESHOLD is 0.70 (exactly 70%) — case-insensitive string match for fill-in-blank answers.
- A `UserProgress` row with `class_id=NULL` is the canonical marker for "module quiz passed."
- Cache TTL of 2 hours (`_QUIZ_CACHE_TTL = 7200`) is long enough for a sitting but short enough to allow regeneration on retry.
- Module quiz only calls `_update_weak_points()` (decreases scores on wrong answers); it never calls `_improve_weak_points()`. Score recovery is reserved for the standalone adaptive quiz — see Section 4.6 for design rationale.

**Failure modes / edge cases handled:**
- Quiz cache expired between GET and POST → regenerates and scores against new questions (user could get different correct answers; logged as warning).
- Already passed → returns `already_passed=True` with empty questions list.
- Gemini returns < 5 questions → raises `RuntimeError` with descriptive message.

**Dependencies on other features:** Course Content Viewing (3.3), Weak Points tracking (4.6), Journey (4.2 — completion hook), Gamification (4.3 — XP on pass), Evaluation Feedback survey (7.14).

### 3.5 Adaptive Standalone Quiz (Weak Points + BPS Recalc)

**One-line purpose:** Generate a personalized 15-question quiz targeting the user's identified weak areas, and recalculate their BPS proficiency level after submission.

**User-facing trigger:** User navigates to `/quiz/adaptive` and clicks "Start Quiz".

**Full file chain (in execution order):**

7. `submit_standalone_quiz()`: scores; saves `StandaloneQuizAttempt`; calls `_update_weak_points()` (wrong) + `_improve_weak_points()` (correct); calls `_calculate_cefr_level()` → updates `users.proficiency_level`

- `_calculate_cefr_level()` in `backend/services/quiz_service.py` (function name is legacy; returns BPS labels)
- `_update_weak_points()`, `_improve_weak_points()` in `backend/services/quiz_service.py`
- `check_bps_change()` in `backend/services/journey_service.py`
- `_generate_and_save_milestone_card()` in `backend/services/quiz_service.py`

**External services called:** Google Gemini API (quiz generation + milestone card image)

**Key technical decisions:**
- `_calculate_cefr_level()` averages the last 3 standalone quiz scores: ≥80% → BPS-4, ≥60% → BPS-3, ≥40% → BPS-2, else BPS-1. Function is named "cefr" internally but returns BPS labels — legacy naming convention.

- No vocabulary/grammar history → falls back to "general beginner Malay questions" (Gemini still generates sensible questions).

### 4.6 Weak Points Tracking

**One-line purpose:** Maintain a per-user topic-level mastery score (0.0–1.0) that drives adaptive quiz personalization.

**User-facing trigger:** Background/automatic — after any quiz submission.

**Full file chain (in execution order):**
1. Module quiz submission → `_update_weak_points(user_id, wrong_questions, db)` only (wrong answers decrease score)
2. Standalone quiz submission → `_update_weak_points()` for wrong answers AND `_improve_weak_points()` for correct answers on existing weak points
3. `_update_weak_points`: upsert `WeakPoint` row — existing: `strength_score -= 0.1` (floor 0.0); new: inserted with `strength_score=0.4`
4. `_improve_weak_points`: existing weak point answered correctly → `strength_score += 0.1` (ceiling 1.0)
5. Adaptive quiz reads top 5 weakest (`ORDER BY strength_score ASC LIMIT 5`)
6. Journey reads top 8 weakest for roadmap prompt personalization

**Key functions/methods involved:**
- `_update_weak_points()` in `backend/services/quiz_service.py` — called from both `submit_module_quiz` and `submit_standalone_quiz`
- `_improve_weak_points()` in `backend/services/quiz_service.py` — called from `submit_standalone_quiz` **only**

**Database tables read/written:** `weak_points` (read + write)

**Key technical decisions:**
- Topic label is the full question text truncated to 200 chars — this is a pragmatic choice; it's not a semantic topic taxonomy.
- Initial strength on first wrong answer is 0.4 (not 0.0) — avoids over-penalizing a single mistake on a new topic.
- **Asymmetric score adjustment by quiz type:** `_improve_weak_points` is only called from the standalone adaptive quiz, never from the module quiz. This is intentional — module quizzes test freshly-taught content with a 70% pass gate; weak point *recovery* is tracked exclusively via the standalone quiz, which is purpose-built to retest historical weaknesses over time.

**Dependencies on other features:** Module Quiz (3.4), Standalone Quiz (3.5), Adaptive Quiz generation, Journey (4.2).

5. Word returned with optional IPA (extracted from meaning string via regex `/ipa/` pattern)
6. User types answer → `gamesApi.submitSpellingAnswer()` → `POST /api/games/spelling/submit`
7. `backend/services/spelling_service.py` — `evaluate_answer()`: Levenshtein distance — 0 = correct (+2 XP), 1 = almost, ≥2 = wrong
8. On session end → `POST /api/games/spelling/session` → `save_session_score()` — only saves if better than today's existing score

**Key functions/methods involved:**
- `get_next_word()`, `evaluate_answer()`, `save_session_score()`, `get_personal_best()` in `backend/services/spelling_service.py`
- `_levenshtein(a, b)` — O(m×n) DP implementation
- `_extract_ipa(meaning)` — regex for `/ipa/` pattern in meaning string

**Database tables read/written:** `vocabulary_learned` (read), `spelling_game_scores` (read + write), `users` (XP write via gamification)

**Key technical decisions:**
- Words answered incorrectly get 3× selection weight — Leitner-inspired spaced repetition without a full SM-2 implementation.
- Levenshtein edit-distance 1 is "almost" not "wrong" — improves UX for minor typos (e.g., missing diacritic).
- Only today's best session score is retained (not all sessions) — keeps the `spelling_game_scores` table small.

## 8. Architecture Patterns I Should Recognize

1. **Router → Service layering:** All business logic lives in `backend/services/`, never in `backend/routers/`. Routers are thin wrappers that validate input (via Pydantic), call the service, and return the response. No raw DB queries or Gemini calls in routers.

2. **Fire-and-forget background tasks with strong references:** `asyncio.create_task()` is used extensively for non-blocking post-response work (vocab extraction, XP awards, milestone image generation). `journey_service.py` stores tasks in a module-level set `_background_tasks` with a `done_callback` to prevent premature GC — a documented asyncio gotcha.

3. **Three-layer Redis cache for chatbot:** Every chatbot request checks three Redis keys in parallel (`asyncio.gather`): RAG context cache, user profile cache, and conversation history cache. DB fallback is sequential (safe because sessions aren't concurrent). This saves 10-15ms on the hot path.

4. **Gemini retry with 429-aware backoff:** `_invoke_with_retry()` parses the suggested delay from 429 error messages (`retry in Xs`) and waits that exact amount + 2s buffer. For other errors, uses exponential backoff (1s, 2s, 4s). This is more respectful of Gemini's rate limit signals than blind exponential backoff.

5. **Semaphore for concurrent Gemini calls:** `asyncio.Semaphore(2)` in `course_service.py` limits parallel class-content generation to 2 concurrent requests. This is the key mechanism for respecting the 5 RPM free-tier limit on Gemini 2.5 Flash.

6. **Template/clone deduplication for courses:** First generation of a topic+level creates a template (`is_template=True`). Subsequent users with the same topic+level get a clone — zero AI calls, instant delivery. The template slug is normalized to handle minor phrasing variations.

7. **Correct answers never leave the server:** Quiz questions are generated and cached on the server WITH `correct_answer`. They are stripped before sending to the frontend. Scoring is done entirely server-side by loading the cached questions. This prevents answer inspection via browser dev tools.

8. **BPS proficiency is a rolling average, not a snapshot:** `_calculate_cefr_level()` averages the last 3 standalone quiz attempts. This prevents a single lucky/unlucky quiz from dramatically changing the displayed level.

9. **Dual-TTL strategy for profile vs. history:** User profile cache TTL is 5 minutes (changes infrequently). Conversation history cache TTL is 30 minutes (changes every message). After each assistant message, history cache is updated in-place without a DB round-trip (in `_update_history_cache()`).

10. **Identity-verified destructive actions:** Both user account deletion (`profile/delete-account`) and roadmap deletion (`journey/roadmap/verify-and-delete`) require re-verification of identity (password for email accounts; email confirmation for Google accounts). Admin destructive actions (delete/reset user) require the admin's own password.

## 11. Decisions an Examiner Will Probe

**Q: Why pgvector instead of a dedicated vector database like Pinecone or Weaviate?**  
A: BahasaBot's RAG corpus is small (~hundreds of Malay language knowledge chunks) and usage is ~30 concurrent users. pgvector avoids a separate service, reduces infrastructure cost to zero (Neon supports it natively), keeps the stack simpler (one database for everything), and cosine similarity search at this scale adds negligible latency. Dedicated vector databases are justified at millions of embeddings or high query rates.

**Q: Why dual JWT (access + refresh) instead of a single long-lived token?**  
A: Short access tokens (15 min) limit the window of compromise if a token is stolen. The refresh token (7 days) allows seamless session continuation without requiring re-login. This is the industry-standard pattern. A single long-lived token would extend the compromise window to days.

**Q: Why `asyncio.Semaphore(2)` in course generation?**  
A: Gemini 2.5 Flash free tier has a 5 RPM limit. Course generation makes N concurrent class-content calls (N = number of classes across all modules). Without the semaphore, all calls fire simultaneously and 429 errors cascade. Semaphore(2) ensures at most 2 concurrent Gemini calls, respecting the rate limit while still parallelizing generation.

**Q: Why SSE instead of WebSockets for chatbot streaming?**  
A: SSE is simpler (standard HTTP, no upgrade handshake), sufficient for one-directional streaming (server → client), works over HTTP/2 multiplexing, and requires less client-side code. WebSockets add bidirectional capability not needed here, plus more complex state management and deployment considerations (Railway/Render WebSocket support variability).

**Q: Why is correct_answer never sent to the frontend?**  
A: Server-side scoring prevents cheating — users cannot inspect browser dev tools or network requests to find correct answers before submitting. The cache holds the full questions; the GET endpoint strips `correct_answer`; the POST endpoint loads from cache for scoring. This is a deliberate security design.

**Q: Why `bcrypt` directly instead of `passlib`?**  
A: `passlib` has a known bug with `bcrypt >= 4.x` (the version compatibility check fails, causing 500 errors on all password operations). Using `bcrypt.hashpw()` and `bcrypt.checkpw()` directly avoids this dependency issue entirely.

**Q: Why `statement_cache_size=0` in the database engine?**  
A: Neon's pooler endpoint runs PgBouncer in transaction mode. PgBouncer returns connections to its pool after each transaction, meaning the same asyncpg connection may get a different underlying PostgreSQL server connection on the next request. Asyncpg caches prepared statements per connection — those cached statements no longer exist on the new server connection. Setting cache size to 0 disables this caching, preventing `"prepared statement does not exist"` errors that would otherwise appear ~5 minutes after the app starts under load.

**Q: Why BPS (BahasaBot Proficiency Scale) instead of CEFR?**  
A: CEFR levels (A1, A2, B1, B2) are calculated by the system based on quiz performance — they're not externally validated certifications. Using a project-branded scale (BPS-1 through BPS-4) is more academically honest and avoids implying CEFR compliance. The mapping is 1:1 but clearly scoped to this platform's measurement methodology.

**Q: Why a template/clone system for courses?**  
A: Without deduplication, two users generating "ordering food at a restaurant" would make identical Gemini API calls, burning tokens and producing nearly identical content. The template system generates once per topic+level combination, then clones for subsequent users. This is a cost optimization and performance improvement — instant delivery for the second user vs. 30+ seconds generation time.

**Q: Why httpx for image generation instead of the Google Generative AI SDK?**  
A: The `google-generativeai` SDK version 0.7.x used by this project does not support `responseModalities` in `GenerationConfig` (required for image-only responses). Attempting to pass it via the SDK causes a silent `TypeError` — no images are generated but no error is raised. Using `httpx` to call the REST API directly bypasses the SDK limitation entirely.

**Q: Why is weak point topic stored as the full question text (not a semantic category)?**  
A: The question text IS the topic at this granularity — e.g., "What is the Malaysian Malay word for 'food'?" tells the adaptive quiz exactly what to re-test. A semantic category taxonomy would require additional AI classification per question, adding latency and complexity. The trade-off is that very similar questions create separate weak point entries; the approach is pragmatic for an FYP scope.

**Q: Why `fuzzywuzzy.fuzz.token_sort_ratio(≥70)` for journey completion matching?**  
A: Users type their own topics when generating courses (e.g., "ordering food at a restaurant") and the journey roadmap has element topics Gemini generated (e.g., "Food ordering and restaurant phrases"). Exact string matching would never connect them. `token_sort_ratio` was chosen over `partial_ratio` because it sorts tokens alphabetically before comparing — making it insensitive to word order differences (e.g., "Basic Malay Greetings" correctly matches "Greetings Basic Malay"). A threshold of 70 is strict enough to avoid false positives between unrelated topics while still tolerating natural rephrasing variations.

**Q: Why does the module quiz only decrease weak point scores but never increase them?**  
A: Module quizzes test content the user has just learned in that specific module — they are pass-gated assessments, not remediation tools. Weak point *recovery* (increasing `strength_score`) is reserved exclusively for the standalone adaptive quiz, which is purpose-built to revisit historical weaknesses across all past learning. This separation keeps the two quiz types conceptually clean: the module quiz measures whether you can proceed; the standalone quiz measures whether you've actually internalized what was hard. Mixing recovery into the module quiz would dilute both purposes.

**Q: Why are journey notification flags stored in Redis without TTL (permanent)?**  
A: The halfway-timeline warning and 7-day deadline warning should fire exactly once per roadmap lifetime. Storing them in Redis without TTL means they persist indefinitely for a given user_id — which is correct, since the flag is per-roadmap (the user_id key is effectively per-active-roadmap because only one active roadmap exists at a time). If the user creates a new roadmap, the old key is irrelevant because the halfway/7-day thresholds are recalculated fresh.

## 12. Things That Are NOT Implemented (Honest Gaps)

1. **`audio_service.py` does not exist.** The CLAUDE.md lists `backend/services/audio_service.py` in the folder structure but the file was not found in the codebase. Pronunciation is handled entirely client-side via the Web Speech API hook (`usePronunciation.ts`) with no backend involvement.

2. **No WebSocket or persistent connection for chatbot.** The chatbot uses SSE (one-directional streaming). If the connection drops mid-stream, the frontend must retry; there is no automatic resume.

3. **No Celery task queue.** Course generation uses FastAPI's built-in `BackgroundTasks` (in-process). For a single-process deployment this works fine for 30 users; it would not survive process restarts or scale to multiple workers without losing job state.

4. **No per-user rate limit tracking in DB.** Rate limits are stored only in Redis. If Redis goes down, rate limiting degrades to in-memory (per-process, not distributed).

5. **No Google OAuth native redirect flow.** Google OAuth is handled by posting the ID token from Google Identity Services (client-side popup) to `/api/auth/google`. There is no server-side OAuth redirect callback — this means Google sign-in requires JavaScript and cannot work without the client-side Google SDK.

6. **Streak is not granularly reset.** If a user misses a day, their streak resets to 1 on next activity. There is no "streak freeze" or grace period mechanism.

7. **No real-time notification push.** Notifications are polled every 60 seconds (`POLL_INTERVAL_MS = 60_000`). New notifications from background tasks (streak milestone, course complete) won't appear until the next poll cycle — up to 1 minute delay.

8. **Stub module quiz route pages.** `frontend/app/(dashboard)/quiz/module/[moduleId]/page.tsx` and its `results/` sibling are placeholder stubs (`TODO: Implement in Phase 5`). The working module quiz is served from the course-scoped path: `frontend/app/(dashboard)/courses/[courseId]/modules/[moduleId]/quiz/`. The stub files exist in the repo but are never reachable via the application's navigation.

9. **Password reset tokens are not cleaned up.** Used and expired `password_reset_tokens` rows are never deleted — they accumulate in the table. A periodic cleanup job or DB-level TTL would be needed in production.

10. **RAG corpus is static.** `MALAY_CORPUS` from `backend/data/malay_corpus.py` is seeded once on app startup if the documents table is empty. There is no admin interface to add/update corpus chunks without a code deploy.

11. **No mobile app.** The platform is web-only (Next.js). The sidebar is responsive but there is no native iOS/Android app.

12. **No email verification on registration.** Users can register with any email address without verifying ownership. This could allow account creation with someone else's email.
